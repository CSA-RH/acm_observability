
# User Workload Monitoring with Standard Metrics Collection

This document covers integrating OpenShift User Workload Monitoring (UWM) with ACM observability **using the standard metrics-collector**. It explains how to enable UWM on managed clusters, deploy a sample application with a ServiceMonitor, forward user workload metrics to the hub's observability, and create hub-side dashboards and alerts based on application-specific metrics.

> **This document applies to the standard metrics collection stack** (ConfigMap-based allowlists + `metrics-collector`). The standard and MCOA stacks are mutually exclusive — only one can be active at a time.

## User workload Monitoring (UWM) with ACM

UWM runs in each OpenShift cluster, in a separate Prometheus instance (`prometheus-user-workload`), that scrapes metrics from user workloads namespaces via `ServiceMonitor` and `PodMonitor` CRs. 
- A `ServiceMonitor` selects Kubernetes Services by label and tells Prometheus which port and path to scrape: use it when your application exposes metrics through a Service. 
- A `PodMonitor` selects Pods directly by label, bypassing the need for a Service: use it for sidecar exporters, batch jobs, or pods that expose metrics but have no Service. 

The user workload metrics can be forwarded to the hub, making them available for centralized dashboarding and alerting.

Once on the hub, UWM metrics are handled identically to platform metrics: same Thanos store, same Grafana, same alert/recording rules.

## 1. Enable User Workload Monitoring on the managed clusters

Apply policy on the Hub, to enable the UWM in all the managed clusters:

```bash
oc apply -f - <<'EOF'
apiVersion: policy.open-cluster-management.io/v1
kind: Policy
metadata:
  name: policy-enable-uwm
  namespace: open-cluster-management-global-set
  annotations:
    policy.open-cluster-management.io/standards: NIST SP 800-53
    policy.open-cluster-management.io/categories: CM Configuration Management
    policy.open-cluster-management.io/controls: CM-2 Baseline Configuration
spec:
  remediationAction: enforce
  disabled: false
  policy-templates:
    - objectDefinition:
        apiVersion: policy.open-cluster-management.io/v1
        kind: ConfigurationPolicy
        metadata:
          name: enable-uwm
        spec:
          remediationAction: enforce
          severity: low
          object-templates:
            - complianceType: musthave
              objectDefinition:
                apiVersion: v1
                kind: ConfigMap
                metadata:
                  name: cluster-monitoring-config
                  namespace: openshift-monitoring
                data:
                  config.yaml: |
                    enableUserWorkload: true
---
apiVersion: cluster.open-cluster-management.io/v1beta1
kind: Placement
metadata:
  name: placement-enable-uwm
  namespace: open-cluster-management-global-set
spec:
  tolerations:
    - key: cluster.open-cluster-management.io/unavailable
      operator: Exists
    - key: cluster.open-cluster-management.io/unreachable
      operator: Exists
  predicates:
    - requiredClusterSelector:
        labelSelector:
          matchLabels:
            vendor: OpenShift
---
apiVersion: policy.open-cluster-management.io/v1
kind: PlacementBinding
metadata:
  name: binding-enable-uwm
  namespace: open-cluster-management-global-set
placementRef:
  name: placement-enable-uwm
  apiGroup: cluster.open-cluster-management.io
  kind: Placement
subjects:
  - name: policy-enable-uwm
    apiGroup: policy.open-cluster-management.io
    kind: Policy
EOF
```

# Verify that the policy is compliant:
```bash
oc get policy -n open-cluster-management-global-set -o custom-columns='NAME:.metadata.name,COMPLIANCE:.status.compliant'
```

## 2. Deploy a sample user workload that exposes metrics

Deploy a simple application in the Hub cluster that exposes Prometheus metrics. The app (`prometheus-example-app`) serves an HTTP endpoint on port 8080 and exposes a `/metrics` path with the following metrics:

| Metric | Type | Description |
|---|---|---|
| `http_requests_total` | Counter | Total number of HTTP requests received, labeled by `method` and `code` |
| `http_request_duration_seconds` | Histogram | Duration of HTTP requests in seconds (with `_bucket`, `_count`, `_sum` suffixes) |
| `version` | Gauge | Application version info |

The metrics are exposed via the standard Prometheus exposition format at `http://<pod-ip>:8080/metrics`. A `ServiceMonitor` CR is created alongside the app to tell `prometheus-user-workload` to scrape this endpoint every 30 seconds:

### 2.1.Fetch the `kubeconfig` from the manged cluster `cluster1`
To allow the manifests to be created in the managed cluster named `cluster1`.

```bash
CLUSTER_NAME=cluster1
SECRET_NAME=$(oc get secret -n $CLUSTER_NAME -o name | grep 'admin-kubeconfig')
oc extract $SECRET_NAME -n $CLUSTER_NAME --keys=kubeconfig --to=- > /tmp/${CLUSTER_NAME}-kubeconfig

# Verify connectivity with managed cluster:
oc --kubeconfig=/tmp/${CLUSTER_NAME}-kubeconfig get nodes
```

### 2.2.Create the namespace and deployment in the managed cluster. 

```bash
oc --kubeconfig=/tmp/${CLUSTER_NAME}-kubeconfig new-project demo-app

oc --kubeconfig=/tmp/${CLUSTER_NAME}-kubeconfig apply -f - <<'EOF'
apiVersion: apps/v1
kind: Deployment
metadata:
  name: demo-metrics-app
  namespace: demo-app
  labels:
    app: demo-metrics-app
spec:
  replicas: 1
  selector:
    matchLabels:
      app: demo-metrics-app
  template:
    metadata:
      labels:
        app: demo-metrics-app
    spec:
      containers:
        - name: metrics-exporter
          image: quay.io/brancz/prometheus-example-app:v0.5.0
          ports:
            - name: http-metrics
              containerPort: 8080
---
apiVersion: v1
kind: Service
metadata:
  name: demo-metrics-app
  namespace: demo-app
  labels:
    app: demo-metrics-app
spec:
  ports:
    - name: http-metrics
      port: 8080
      targetPort: 8080
  selector:
    app: demo-metrics-app
---
apiVersion: monitoring.coreos.com/v1
kind: ServiceMonitor
metadata:
  name: demo-metrics-app
  namespace: demo-app
spec:
  endpoints:
    - port: http-metrics
      interval: 30s
  selector:
    matchLabels:
      app: demo-metrics-app
EOF

oc --kubeconfig=/tmp/${CLUSTER_NAME}-kubeconfig expose svc demo-metrics-app
```

Verify the metrics are scraped by the managed cluster UWM - it can take 1/2 minutes to show metrics:

```bash
oc --kubeconfig=/tmp/${CLUSTER_NAME}-kubeconfig -n openshift-user-workload-monitoring exec -it prometheus-user-workload-0 -- \
  promtool query instant http://localhost:9090 'version{namespace="demo-app"}'
```

The sample app exposes metrics like `version`, `http_requests_total`, `http_request_duration_seconds_*`.

## 3. Forward user workload metrics to the ACM hub

Add the user workload metrics to the custom allowlist. The ConfigMap must be created in the `demo-app` namespace of the managed cluster. Use an ACM policy to deploy it:

```bash
oc apply -f - <<'EOF'
apiVersion: policy.open-cluster-management.io/v1
kind: Policy
metadata:
  name: policy-uwm-allowlist
  namespace: open-cluster-management-global-set
  annotations:
    policy.open-cluster-management.io/standards: NIST SP 800-53
    policy.open-cluster-management.io/categories: CM Configuration Management
    policy.open-cluster-management.io/controls: CM-2 Baseline Configuration
spec:
  remediationAction: enforce
  disabled: false
  policy-templates:
    - objectDefinition:
        apiVersion: policy.open-cluster-management.io/v1
        kind: ConfigurationPolicy
        metadata:
          name: uwm-allowlist
        spec:
          remediationAction: enforce
          severity: low
          object-templates:
            - complianceType: musthave
              objectDefinition:
                apiVersion: v1
                kind: ConfigMap
                metadata:
                  name: observability-metrics-custom-allowlist
                  namespace: demo-app
                data:
                  uwl_metrics_list.yaml: |
                    names:
                      - http_requests_total
                      - http_request_duration_seconds_bucket
                      - http_request_duration_seconds_count
                      - http_request_duration_seconds_sum
                      - version
---
apiVersion: policy.open-cluster-management.io/v1
kind: PlacementBinding
metadata:
  name: binding-uwm-allowlist
  namespace: open-cluster-management-global-set
placementRef:
  name: placement-uwm-allowlist
  apiGroup: cluster.open-cluster-management.io
  kind: Placement
subjects:
  - name: policy-uwm-allowlist
    apiGroup: policy.open-cluster-management.io
    kind: Policy
---
apiVersion: cluster.open-cluster-management.io/v1beta1
kind: Placement
metadata:
  name: placement-uwm-allowlist
  namespace: open-cluster-management-global-set
spec:
  predicates:
    - requiredClusterSelector:
        labelSelector:
          matchLabels:
            name: cluster1
EOF
```

Verify that the policy is compliant:

```bash
oc get policy -n open-cluster-management-global-set -o custom-columns='NAME:.metadata.name,COMPLIANCE:.status.compliant'
```

Wait for the metrics-collector to pick up the new allowlist and forward the metrics to the hub. The metrics-collector pod restarts when the ConfigMap changes, and then collects and forwards data on a ~5-minute interval (300s). Allow **up to 5 minutes** after the restart for the first data to appear in Grafana.

## 4. Generate Traffic

```bash
ROUTE=$(oc --kubeconfig=/tmp/${CLUSTER_NAME}-kubeconfig -n demo-app get route demo-metrics-app -o jsonpath='{.spec.host}')
for i in $(seq 1 30); do curl -s "http://$ROUTE/" > /dev/null; done
echo "Sent 30 requests"
```

## 5. Verify user workload metrics on the hub Grafana

```bash
# Search for: 
http_requests_total{namespace="demo-app"}
```
