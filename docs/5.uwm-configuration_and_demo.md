
# User Workload Monitoring in the ACM Context

This document covers integrating OpenShift User Workload Monitoring (UWM) with ACM observability. It explains how to enable UWM on managed clusters, deploy a sample application with a ServiceMonitor, forward user workload metrics to the hub's Thanos, and create hub-side dashboards and alerts based on application-specific metrics.

## User workload Monitoring (UWM) with ACM

UWM runs a separate Prometheus instance (`prometheus-user-workload`) that scrapes metrics from user namespaces via `ServiceMonitor` and `PodMonitor` CRs. A `ServiceMonitor` selects Kubernetes Services by label and tells Prometheus which port and path to scrape : use it when your application exposes metrics through a Service. A `PodMonitor` selects Pods directly by label, bypassing the need for a Service : use it for sidecar exporters, batch jobs, or pods that expose metrics but have no Service. ACM's observability collector can forward these user workload metrics to the hub's Thanos, making them available for centralized dashboarding and alerting.

Once on the hub, UWM metrics are handled identically to platform metrics : same Thanos store, same Grafana, same alert/recording rules:

| Aspect | Platform Metrics | User Workload Metrics |
|---|---|---|
| **Source Prometheus** | `prometheus-k8s` in `openshift-monitoring` | `prometheus-user-workload` in `openshift-user-workload-monitoring` |
| **Discovery** | Built-in OpenShift scrape targets | `ServiceMonitor` / `PodMonitor` CRs in user namespaces |
| **Allowlist ConfigMap key** | `metrics_list.yaml` | `uwl_metrics_list.yaml` |
| **ConfigMap location** | Hub (`open-cluster-management-observability`) : propagated to all clusters | Managed cluster (workload namespace or `open-cluster-management-addon-observability`) |
| **Destination on hub** | Thanos Receive | Same Thanos Receive |
| **Hub-side alerting/recording** | `thanos-ruler-custom-rules` | Same `thanos-ruler-custom-rules` |
| **Grafana querying** | Same Grafana, same data source | Same Grafana, same data source |

## 1. Enable User Workload Monitoring on the managed cluster

On each managed cluster (or the hub if self-managed), enable UWM:

```bash
oc apply -f - <<'EOF'
apiVersion: v1
kind: ConfigMap
metadata:
  name: cluster-monitoring-config
  namespace: openshift-monitoring
data:
  config.yaml: |
    enableUserWorkload: true
EOF

# Verify UWM pods are running
oc -n openshift-user-workload-monitoring get pods
# Expected: prometheus-user-workload-0, prometheus-user-workload-1, thanos-ruler-user-workload-*
```

## 2. Deploy a sample user workload that exposes metrics

Deploy a simple application, in the Hub cluster, that exposes Prometheus metrics:

```bash
oc new-project demo-app 2>/dev/null || oc project demo-app

oc apply -f - <<'EOF'
apiVersion: apps/v1
kind: Deployment
metadata:
  name: demo-metrics-app
  namespace: demo-app
  labels:
    app: demo-metrics-app
spec:
  replicas: 1
  selector:
    matchLabels:
      app: demo-metrics-app
  template:
    metadata:
      labels:
        app: demo-metrics-app
    spec:
      containers:
        - name: metrics-exporter
          image: quay.io/brancz/prometheus-example-app:v0.5.0
          ports:
            - name: http-metrics
              containerPort: 8080
---
apiVersion: v1
kind: Service
metadata:
  name: demo-metrics-app
  namespace: demo-app
  labels:
    app: demo-metrics-app
spec:
  ports:
    - name: http-metrics
      port: 8080
      targetPort: 8080
  selector:
    app: demo-metrics-app
---
apiVersion: monitoring.coreos.com/v1
kind: ServiceMonitor
metadata:
  name: demo-metrics-app
  namespace: demo-app
spec:
  endpoints:
    - port: http-metrics
      interval: 30s
  selector:
    matchLabels:
      app: demo-metrics-app
EOF

oc expose svc demo-metrics-app
```

Verify the metrics are scraped by UWM:

```bash
# Check that the ServiceMonitor target is up:
# In the OpenShift console: Observe > Targets > Source: User

# Or query locally:
oc -n openshift-user-workload-monitoring exec -it prometheus-user-workload-0 -- \
  promtool query instant http://localhost:9090 'version{namespace="demo-app"}'
```

The sample app exposes metrics like `version`, `http_requests_total`, `http_request_duration_seconds_*`.

## 3. Forward user workload metrics to the ACM hub

Add the user workload metrics to the custom allowlist. For user workload metrics, create the `observability-metrics-custom-allowlist` ConfigMap **in the namespace of the workload** on the managed cluster (or in `open-cluster-management-addon-observability` for all namespaces):

```bash
# On the managed cluster : forward metrics from demo-app namespace:
oc apply -f - <<'EOF'
kind: ConfigMap
apiVersion: v1
metadata:
  name: observability-metrics-custom-allowlist
  namespace: demo-app
data:
  uwl_metrics_list.yaml: |
    names:
      - http_requests_total
      - http_request_duration_seconds_bucket
      - http_request_duration_seconds_count
      - http_request_duration_seconds_sum
      - version
EOF
```

> **Alternative:** To collect user workload metrics from **all namespaces** on the managed cluster, create the ConfigMap in `open-cluster-management-addon-observability` instead.

Wait for the metrics collector to pick up the new allowlist (~1-2 minutes) and forward the metrics to the hub.

## 4. Verify user workload metrics on the hub

On the hub cluster, query the forwarded metrics in Grafana:

```bash
# Search for: 
http_requests_total{namespace="demo-app"}
```

## 5. Create an alert for the user workload

Add an alert to the hub that fires when the demo-app request error rate exceeds a threshold:

Update the `thanos-ruler-custom-rules` ConfigMap (add to existing groups):

```bash
oc apply -f - <<'EOF'
kind: ConfigMap
apiVersion: v1
metadata:
  name: thanos-ruler-custom-rules
  namespace: open-cluster-management-observability
data:
  custom_rules.yaml: |
    groups:
      - name: acm-demo-user-workload-alerts
        rules:
        - alert: DemoAppHighRequestRate
          annotations:
            summary: "demo-app on {{ $labels.cluster }} has high request rate"
            description: "demo-app on cluster {{ $labels.cluster }} received more than 5 requests in the last 10 minutes."
          expr: |
            sum by (cluster) (increase(http_requests_total{namespace="demo-app"}[10m])) > 5
          for: 5m
          labels:
            severity: warning
            source: acm-demo-user-workload
EOF
```

## 6. Generate Alert

```bash
ROUTE=$(oc -n demo-app get route demo-metrics-app -o jsonpath='{.spec.host}')
for i in $(seq 1 50); do curl -s "http://$ROUTE/" > /dev/null; done
echo "Sent 10 requests : alert should fire within ~10 min"
```

## Cleanup

```bash
# Delete Perses dashboards
oc delete persesdashboard acm-demo-cluster-health -n open-cluster-management-observability
oc delete persesdashboard acm-demo-user-workload -n open-cluster-management-observability

# Delete custom rules
oc delete configmap thanos-ruler-custom-rules -n open-cluster-management-observability
oc delete configmap observability-metrics-custom-allowlist -n open-cluster-management-observability

# Delete demo app (on managed cluster)
oc delete project demo-app

# Delete user workload allowlist (on managed cluster)
oc delete configmap observability-metrics-custom-allowlist -n demo-app 2>/dev/null

# Delete Perses datasource
oc delete persesglobaldatasource acm-thanos-querier

# Delete UIPlugin
oc delete uiplugin monitoring

# Delete MCO
oc delete mco observability

# Delete MinIO
oc -n open-cluster-management-observability delete deployment minio
oc -n open-cluster-management-observability delete svc minio
oc -n open-cluster-management-observability delete pvc minio-pvc
oc -n open-cluster-management-observability delete secret thanos-object-storage
oc -n open-cluster-management-observability delete secret multiclusterhub-operator-pull-secret
```
