
# MCO Install & Configuration

> **Disclaimer:** This repository is prepared for **demonstration and educational purposes**. The configurations, thresholds, and examples used here are simplified to illustrate concepts and are not intended for production use. Always review and adapt the configurations to your environment's requirements, security policies, and best practices before applying them to production clusters. Additionally, some components used in these demos (e.g., MinIO) are **not supported or maintained by Red Hat** and are included solely for demonstration convenience.

This document covers the installation and configuration of the MultiClusterObservability (MCO) operator on the hub cluster. It includes deploying MinIO as the S3-compatible object storage backend for Thanos, creating the required secrets (pull secret, object storage config), and applying the `MultiClusterObservability` CR to enable the observability stack. It also covers SNO-specific prerequisites (increasing the max pods limit).

---

## 1. Install and Configure MCO with MCOA + MinIO

### 1.0 If using a SNO, increase allowed pods to 500.

I used a SingleNodeOpenShift sandbox cluster for the testings, therefore it was required to increase the number of allowed pods to 500, to allow more than 250 to be schedualed into the cluster.

1.0.1.Get the labels of the mc
oc get machineconfigpool worker -o yaml | grep 'labels:' -A4

1.0.2.Create a KubeletConfig, adding the labels from 1.1
Note: If SNO, should be the "master".

```bash
cat << EOF > kubeletconfig.yaml
apiVersion: machineconfiguration.openshift.io/v1
kind: KubeletConfig
metadata:
  name: set-max-pods 
spec:
  machineConfigPoolSelector:
    matchLabels:
      pools.operator.machineconfiguration.openshift.io/master: "" 
  kubeletConfig:
    podsPerCore: 10 
    maxPods: 500
EOF

oc apply -f kubeletconfig.yaml
```

Applying this configuration will automatically execute a restart of the node, to apply the configuration.

Check that the node is configured with 500 pods
```bash
oc get node <node-name> -o jsonpath='{.status.allocatable.pods}'
```

### 1.1 Create the observability namespace

```bash
oc create namespace open-cluster-management-observability
```

### 1.2 Deploy MinIO as object storage

MinIO provides S3-compatible object storage for Thanos. 

create MinIO manually:

```bash
oc apply -f - <<'EOF'
---
apiVersion: v1
kind: Namespace
metadata:
  name: minio
---
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: minio-pvc
  namespace: minio
spec:
  accessModes:
    - ReadWriteOnce
  resources:
    requests:
      storage: 10Gi
---
apiVersion: apps/v1
kind: Deployment
metadata:
  name: minio
  namespace: minio
spec:
  replicas: 1
  selector:
    matchLabels:
      app: minio
  template:
    metadata:
      labels:
        app: minio
    spec:
      containers:
        - name: minio
          image: quay.io/minio/minio:latest
          args:
            - server
            - /data
            - --console-address
            - ":9001"
          env:
            - name: MINIO_ROOT_USER
              value: minio
            - name: MINIO_ROOT_PASSWORD
              value: minio123
          ports:
            - containerPort: 9000
            - containerPort: 9001
          volumeMounts:
            - name: data
              mountPath: /data
      volumes:
        - name: data
          persistentVolumeClaim:
            claimName: minio-pvc
---
apiVersion: v1
kind: Service
metadata:
  name: minio
  namespace: minio
spec:
  ports:
    - name: api
      port: 9000
      targetPort: 9000
    - name: console
      port: 9001
      targetPort: 9001
  selector:
    app: minio
EOF
```

Wait for MinIO to be ready:

```bash
oc -n minio wait --for=condition=Available deployment/minio --timeout=120s
```

Create the Thanos bucket in MinIO. The MinIO container image includes its own `mc` binary, so exec into the pod to avoid installing anything locally:

```bash
oc -n minio exec deployment/minio -- mc alias set local http://localhost:9000 minio minio123
oc -n minio exec deployment/minio -- mc mb local/thanos
```

### 1.3 Create the Thanos object storage secret

```bash
cat << EOF > thanos-object-storage.yaml
apiVersion: v1
kind: Secret
metadata:
  name: thanos-object-storage
  namespace: open-cluster-management-observability
type: Opaque
stringData:
  thanos.yaml: |
    type: s3
    config:
      bucket: thanos
      endpoint: minio.minio.svc:9000
      insecure: true
      access_key: minio
      secret_key: minio123
EOF

oc apply -f thanos-object-storage.yaml
```

Verify:

```bash
oc -n open-cluster-management-observability get secret thanos-object-storage -o jsonpath='{.data.thanos\.yaml}' | base64 -d
```

### 1.4 Copy the pull secret

MCO requires the `multiclusterhub-operator-pull-secret` in the observability namespace:

```bash
DOCKER_CONFIG_JSON=$(oc extract secret/pull-secret -n openshift-config --to=-)
oc -n open-cluster-management-observability create secret generic multiclusterhub-operator-pull-secret \
  --from-literal=.dockerconfigjson="$DOCKER_CONFIG_JSON" \
  --type=kubernetes.io/dockerconfigjson
```

### 1.5 Create the MultiClusterObservability CR

ACM supports two **mutually exclusive** metrics collection stacks. Only one can be active at a time, enabling MCOA automatically disables the standard collector, and vice versa.

Both stacks share the same hub-side components (Thanos Receive, Thanos Ruler, Alertmanager, Grafana) and the same `thanos-ruler-custom-rules` ConfigMap for hub-side alert/recording rules. The difference is only in how metrics are collected and forwarded from managed clusters.

#### Option 1: Standard metrics-collector

Creating the following `MultiClusterObservability` CR, deploys the standard `metrics-collector` on each managed cluster. No additional steps are needed.

```bash
oc apply -f - <<'EOF'
apiVersion: observability.open-cluster-management.io/v1beta2
kind: MultiClusterObservability
metadata:
  name: observability
  namespace: open-cluster-management-observability
spec:
  observabilityAddonSpec: {}
  storageConfig:
    metricObjectStorage:
      name: thanos-object-storage
      key: thanos.yaml
EOF
```

Wait for the stack to deploy:

```bash
# Watch all pods come up (Thanos, Grafana, AlertManager, etc.)
oc -n open-cluster-management-observability get pods -w

# Check the MCO status
oc get mco observability -o jsonpath='{.status.conditions[?(@.type=="Ready")].message}'
# Expected: "Observability components are deployed and running"
```

#### Option 2: MCOA (Multicluster Observability Add-on)

MCOA is not active by default. After the MCO CR is created (Option 1 above), enable MCOA by patching it. This **stops** the standard `metrics-collector` and deploys the `multicluster-observability-addon-manager` + `PrometheusAgent` instead.

```bash
#If MCO already exists, just patch it:
oc patch mco observability -n open-cluster-management-observability --type=merge -p \
  '{"spec":{"capabilities":{"platform":{"metrics":{"default":{"enabled": true}}},"userWorkloads":{"metrics":{"default":{"enabled": true}}}}}}'
```

Verify MCOA is active:

```bash
# The addon-manager should be running:
oc get pods -n open-cluster-management-observability | grep addon-manager

# Default PrometheusAgent and ScrapeConfig resources should exist:
oc get prometheusagents -n open-cluster-management-observability
oc get scrapeconfigs.monitoring.rhobs -n open-cluster-management-observability

# Verify the ClusterManagementAddOn has placements:
oc get cma multicluster-observability-addon -o yaml | yq '.spec.installStrategy.placements'
```

### 1.6 Verify metrics are flowing by connecting to the Grafana instance deployed by MCO

```bash
#default Grafana route
oc -n open-cluster-management-observability get route | grep grafana-open-cluster-management-observability
```

## 2. Connecting to a managed cluster from the hub

If the managed cluster was provisioned by Hive (e.g., via ACM), the admin kubeconfig is stored as a secret in the cluster's namespace on the hub. Extract it with:

```bash
CLUSTER_NAME=cluster1
SECRET_NAME=$(oc get secret -n $CLUSTER_NAME -o name | grep 'admin-kubeconfig')
oc extract $SECRET_NAME -n $CLUSTER_NAME --keys=kubeconfig --to=- > /tmp/${CLUSTER_NAME}-kubeconfig

# Verify connectivity:
oc --kubeconfig=/tmp/${CLUSTER_NAME}-kubeconfig get nodes
```
