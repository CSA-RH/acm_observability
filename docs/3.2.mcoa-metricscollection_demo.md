
# ACM Observability - MCOA metrics collection: Custom Alerts, Recording Rules, and Metrics

This document covers the **Multicluster Observability Add-on (MCOA)**, the new standard for metrics collection in ACM. For the standard stack (ConfigMap-based allowlists + `metrics-collector`), see [3.1 : Standard stack](3.1.standart-metricscollection_demo.md).

---

## 1. Concepts

### 1.1 What is MCOA and how it differs from the standard stack

MCOA changes how **managed (spoke) clusters** collect and forward metrics to the hub. It replaces the `metrics-collector` with a standard `PrometheusAgent`, managed by the Cluster Observability Operator. Instead of ConfigMap-based allowlists, metrics selection and recording rules are defined through standard, Kubernetes/Prometheus CRDs (`ScrapeConfig`, `PrometheusRule`) deployed on each spoke cluster. The hub-side components (Thanos Receive, Thanos Ruler, Alertmanager, `thanos-ruler-custom-rules` ConfigMap) remain unchanged.


### 1.2 Data flow: how metrics move from managed clusters to the hub (MCOA)

```
Managed Cluster                              Hub Cluster
┌──────────────────────┐                     ┌──────────────────────────────────┐
│                      │                     │                                  │
│  Prometheus          │                     │  Observatorium API               │
│  (platform + UWM)    │                     │  (gateway)                       │
│        │             │                     │        │                         │
│        │ /federate   │                     │        ▼                         │
│        ▼             │                     │  Thanos Receive                  │
│  PrometheusAgent     │──── push metric ───>│  (ingests metrics)               │
│  (scrapes via        │                     │        │                         │
│   ScrapeConfig CRs)  │                     │        ▼                         │
│                      │                     │  Thanos Store / Query            │
│  PrometheusRule CRs  │                     │  (long-term storage + queries)   │
│  (local recording    │                     │        │                         │
│   rules evaluated    │                     │        ▼                         │
│   by Prometheus)     │                     │  Thanos Ruler                    │
│                      │                     │  (evaluates hub-side alert &     │
│  Alertmanager        │                     │   recording rules from           │
│  (NOT automatic —    │── fired alerts ────>│   thanos-ruler-custom-rules)     │
│   requires Policy)   │   (if configured)   │        │                         │
│                      │                     │        ▼                         │
│                      │                     │  Alertmanager                    │
│                      │                     │  (deduplicates, routes, silences)│
│                      │                     │        │                         │
│                      │                     │        ▼                         │
│                      │                     │  Grafana                         │
│                      │                     │  (dashboards + explore)          │
│                      │                     │                                  │
└──────────────────────┘                     └──────────────────────────────────┘
```

Metrics and Alerts:
- **Metrics** : The `PrometheusAgent` on each managed cluster federates metrics from the local Prometheus (filtered by `ScrapeConfig` CRs) and remote-writes them to the hub's Thanos Receive. Default `ScrapeConfig` resources are auto-generated for platform dashboards. Additionally, `PrometheusRule` CRs can be deployed on managed clusters to define local recording rules, whose computed results are also forwarded to the hub.
- **Alerts** : **Not forwarded by default.** Must be explicitly configured via ACM governance policies (see section 2.2).

ScrapeConfig and PrometheusRule CRs:
- ScrapeConfig tells the PrometheusAgent which metrics to collect from the managed cluster and send to the hub. It's the filter: "out of all the metrics available on the local Prometheus, forward these ones."
- PrometheusRule tells the local Prometheus on the managed cluster to compute something before sending it. Instead of forwarding raw data, it runs a PromQL expression locally and sends the result.

## 2. Notes and Gotchas

### 2.1 MCOA is disabled by default

Unlike the standard stack (which is active as soon as the `MultiClusterObservability` CR exists), MCOA must be explicitly enabled by patching the MCO CR:

```bash
oc patch mco observability -n open-cluster-management-observability --type=merge -p \
  '{"spec":{"capabilities":{"platform":{"metrics":{"default":{"enabled": true}}},"userWorkloads":{"metrics":{"default":{"enabled": true}}}}}}'
```

When enabled, the MCO operator **stops deploying the standard `metrics-collector`** and deploys the `multicluster-observability-addon-manager` instead. The standard and MCOA stacks are mutually exclusive : enabling MCOA replaces the standard collector.

Source: [ACM 2.16 : Enabling the multicluster observability add-on](https://docs.redhat.com/en/documentation/red_hat_advanced_cluster_management_for_kubernetes/2.16/html-single/observability/index#enabling-mcoa)

### 2.2 Alert forwarding is NOT automatic

In the standard stack, platform alerts are forwarded to the hub's Alertmanager automatically. With MCOA, **you must create ACM governance policies** to configure `additionalAlertmanagerConfigs` on each managed cluster. Two policies are needed:

- `mcoa-alert-forward-platform` : for platform alerts (from `openshift-monitoring`)
- `mcoa-alert-forward-uwl` : for user workload alerts (from `openshift-user-workload-monitoring`)

These policies inject the hub's Alertmanager endpoint into the managed cluster's `cluster-monitoring-config` ConfigMap via `object-templates-raw` with hub Go templating. Source: [ACM 2.16 : Forwarding alerts for MCOA](https://docs.redhat.com/en/documentation/red_hat_advanced_cluster_management_for_kubernetes/2.16/html-single/observability/index#forward-alert-mcoa).

### 2.3 COO is a prerequisite

MCOA requires the **Cluster Observability Operator (COO)** to be installed on the hub cluster (and it will be deployed on managed clusters if not already present). The COO provides the `obo-prometheus-operator` that reconciles the `PrometheusAgent`, `ScrapeConfig`, and related CRDs.

### 2.4 ScrapeConfig labels determine the collector

Each custom `ScrapeConfig` must be labeled to indicate which collector it targets:

- `app.kubernetes.io/component: platform-metrics-collector` : federates from the platform Prometheus. The add-on automatically sets `scrapeClass` and `targets` for platform federation.
- `app.kubernetes.io/component: user-workload-metrics-collector` : federates from user workload Prometheus. You may need to manually set `scrapeClass` and `staticConfigs`.

### 2.5 ScrapeConfig must be registered in ClusterManagementAddOn placements

Creating a `ScrapeConfig` CR alone is not enough : it must be **referenced in the `ClusterManagementAddOn` placements** to be deployed to managed clusters. Otherwise it stays on the hub and does nothing.

### 2.6 Migration tool (`allowlist-migration`)

The [ACM 2.16 Release Notes](https://docs.redhat.com/en/documentation/red_hat_advanced_cluster_management_for_kubernetes/2.16/html-single/release_notes/index#observability) mention an `allowlist-migration` CLI tool to convert existing `observability-metrics-allowlist` ConfigMaps to `ScrapeConfig` and `PrometheusRule` resources, and reference a "Migrating custom allowlist" documentation section. As of this writing, the referenced documentation section does not exist in the [ACM 2.16 observability guide](https://docs.redhat.com/en/documentation/red_hat_advanced_cluster_management_for_kubernetes/2.16/html-single/observability); however, the community already provides the tool in the upstream repository:

- **Community tool**: [stolostron/allowlist-migration-mcoa](https://github.com/stolostron/allowlist-migration-mcoa) : converts the `observability-metrics-allowlist` custom resource to `ScrapeConfig` and `PrometheusRule` resources.

This tool should be referenced in the official ACM documentation once the "Migrating custom allowlist" section is published.

## 3. Demos

### Prerequisites

- ACM 2.16 hub cluster with the `MultiClusterObservability` CR created and observability enabled
- **Cluster Observability Operator (COO)** installed on the hub cluster
- At least one managed cluster imported (or `local-cluster`)
- MCOA enabled (see section 2.1)
- `oc` CLI access with cluster-admin

Enable MCOA if not already done (enables both platform and user-workload metrics collection via MCOA):

```bash
oc patch mco observability -n open-cluster-management-observability --type=merge -p \
  '{"spec":{"capabilities":{"platform":{"metrics":{"default":{"enabled": true}}},"userWorkloads":{"metrics":{"default":{"enabled": true}}}}}}'
```

Verify MCOA is active:

```bash
# The addon-manager should be running:
oc get pods -n open-cluster-management-observability | grep addon-manager

# Default PrometheusAgent and ScrapeConfig resources should exist:
oc get prometheusagents -n open-cluster-management-observability
oc get scrapeconfigs.monitoring.rhobs -n open-cluster-management-observability

# Verify the ClusterManagementAddOn has placements:
oc get cma multicluster-observability-addon -o yaml | yq '.spec.installStrategy.placements'
```

---

### Demo 1: Custom Alert Rule on the Hub (Thanos Ruler)

This demo is **identical to the standard stack** : hub-side alert rules are still defined in the `thanos-ruler-custom-rules` ConfigMap and evaluated by Thanos Ruler.

```bash
oc apply -f - <<'EOF'
kind: ConfigMap
apiVersion: v1
metadata:
  name: thanos-ruler-custom-rules
  namespace: open-cluster-management-observability
data:
  custom_rules.yaml: |
    groups:
      - name: cluster-health
        rules:
        - alert: ClusterCPUHealth-jb
          annotations:
            summary: Notify when CPU utilization on a cluster is greater than the defined utilization limit
            description: "The cluster has a high CPU usage: {{ $value }} core for {{ $labels.cluster }} {{ $labels.clusterID }}."
          expr: |
            max(cluster:cpu_usage_cores:sum) by (clusterID, cluster, prometheus) > 5
          for: 5s
          labels:
            cluster: "{{ $labels.cluster }}"
            prometheus: "{{ $labels.prometheus }}"
            severity: critical
EOF
```

Verify the rule is loaded : in Grafana Explore, query `ALERTS{alertname="ClusterCPUHealth-jb"}`.

> **No change from standard stack:** Hub-side rules use the same ConfigMap regardless of whether MCOA or the standard collector is active. The hub's Thanos Ruler evaluates rules against whatever metrics are in Thanos Store.

### Demo 2: Custom ScrapeConfig

Create a `ScrapeConfig` CR to forward `kubelet_volume_stats_used_bytes` from managed clusters to the hub.

**Step 1 : Create the ScrapeConfig:**

```bash
oc apply -f - <<'EOF'
apiVersion: monitoring.rhobs/v1alpha1
kind: ScrapeConfig
metadata:
  name: custom-volume-metrics
  namespace: open-cluster-management-observability
  labels:
    app.kubernetes.io/component: platform-metrics-collector
spec:
  jobName: custom-volume-stats
  metricsPath: /federate
  params:
    match[]:
      - '{__name__="kubelet_volume_stats_used_bytes"}'
EOF
```

**Step 2 : Register the ScrapeConfig in the ClusterManagementAddOn placements:**

The `ScrapeConfig` must be referenced in the `ClusterManagementAddOn` so it gets deployed to managed clusters. Patch the add-on to include the new `ScrapeConfig` in the `global` placement:

```bash
# First, inspect current placements:
oc get cma multicluster-observability-addon -o yaml | yq '.spec.installStrategy.placements'

# Add the custom ScrapeConfig to the global placement configs:
oc patch cma multicluster-observability-addon --type=json -p '[
  {"op": "add", "path": "/spec/installStrategy/placements/0/configs/-", "value": {
    "group": "monitoring.rhobs",
    "resource": "scrapeconfigs",
    "name": "custom-volume-metrics",
    "namespace": "open-cluster-management-observability"
  }}
]'
```

**Step 3 : Verify the PrometheusAgent picks up the new ScrapeConfig - and the new metric is available in the Hub:**

# After several minutes, query the raw metric in Grafana:
```promql
kubelet_volume_stats_used_bytes
```

**Expected result: data appears.** The raw metric is now being federated from managed clusters via the new `ScrapeConfig`. This is a prerequisite for Demo 3, which creates a hub-side recording rule that depends on this metric.

### Demo 3: Hub-side Recording Rule (depends on Demo 2)

Add a recording rule to the hub. This computes PV usage per namespace using `kubelet_volume_stats_used_bytes`, which is already being federated thanks to the `ScrapeConfig` created in Demo 2.

The recording rule expression `sum(kubelet_volume_stats_used_bytes) BY (namespace, clusterID, cluster)` sums the used bytes of all PersistentVolumeClaims, grouping the result by namespace and cluster.

```bash
oc apply -f - <<'EOF'
kind: ConfigMap
apiVersion: v1
metadata:
  name: thanos-ruler-custom-rules
  namespace: open-cluster-management-observability
data:
  custom_rules.yaml: |
    groups:
      - name: cluster-health
        rules:
        - alert: ClusterCPUHealth-jb
          annotations:
            summary: Notify when CPU utilization on a cluster is greater than the defined utilization limit
            description: "The cluster has a high CPU usage: {{ $value }} core for {{ $labels.cluster }} {{ $labels.clusterID }}."
          expr: |
            max(cluster:cpu_usage_cores:sum) by (clusterID, cluster, prometheus) > 5
          for: 5s
          labels:
            cluster: "{{ $labels.cluster }}"
            prometheus: "{{ $labels.prometheus }}"
            severity: critical
      - name: namespace-volume-usage
        rules:
        - record: namespace:kubelet_volume_stats_used_bytes:sum
          expr: sum(kubelet_volume_stats_used_bytes) BY (namespace, clusterID, cluster)
EOF
```

Query in Grafana Explore: `namespace:kubelet_volume_stats_used_bytes:sum`

**Expected result: data appears.** Because Demo 2 already configured the `ScrapeConfig` to forward `kubelet_volume_stats_used_bytes`, the hub's has the input data and the recording rule produces results. Without Demo 2, this rule would return no data.

### Demo 4: Managed-cluster-side Recording Rule via PrometheusRule

With MCOA, managed-cluster-side recording rules are defined as standard `PrometheusRule` CRs.

This example computes the API server request duration 90th percentile on each managed cluster:

**Step 1 : Create the PrometheusRule:**

```bash
oc apply -f - <<'EOF'
apiVersion: monitoring.coreos.com/v1
kind: PrometheusRule
metadata:
  name: custom-recording-rules
  namespace: open-cluster-management-observability
  annotations:
    operator.prometheus.io/controller-id: openshift-platform-monitoring/prometheus-operator
spec:
  groups:
    - name: custom-apiserver
      rules:
        - record: apiserver_request_duration_seconds:histogram_quantile_90
          expr: histogram_quantile(0.90, sum(rate(apiserver_request_duration_seconds_bucket{job="apiserver", verb!="WATCH"}[10m])) by (verb, le))
EOF
```

> **Note:** The `operator.prometheus.io/controller-id` annotation is **required** ([ACM 2.16 documentation](https://docs.redhat.com/en/documentation/red_hat_advanced_cluster_management_for_kubernetes/2.16/html-single/observability/index#mcoa)). It determines which Prometheus operator instance on the managed cluster picks up the rule. Use `openshift-platform-monitoring/prometheus-operator` for platform metrics or `openshift-user-workload-monitoring/prometheus-operator` for user workload metrics. Without this annotation, the Prometheus operator on the managed cluster will not evaluate the rule.

**Step 2 : Register the PrometheusRule in the ClusterManagementAddOn placements:**

```bash
oc patch cma multicluster-observability-addon --type=json -p '[
  {"op": "add", "path": "/spec/installStrategy/placements/0/configs/-", "value": {
    "group": "monitoring.coreos.com",
    "resource": "prometheusrules",
    "name": "custom-recording-rules",
    "namespace": "open-cluster-management-observability"
  }}
]'
```

**Step 3 : Create a ScrapeConfig to forward the recording rule result to the hub:**

The `PrometheusRule` in Step 1 computes a new metric (`apiserver_request_duration_seconds:histogram_quantile_90`) on each managed cluster, but by itself it only exists locally. To make it available on the hub's Thanos, you need a `ScrapeConfig` that tells the `PrometheusAgent` to pick up this metric and forward it:

```bash
oc apply -f - <<'EOF'
apiVersion: monitoring.rhobs/v1alpha1
kind: ScrapeConfig
metadata:
  name: custom-recording-rule-metrics
  namespace: open-cluster-management-observability
  labels:
    app.kubernetes.io/component: platform-metrics-collector
spec:
  jobName: custom-recording-rules
  metricsPath: /federate
  params:
    match[]:
      - '{__name__="apiserver_request_duration_seconds:histogram_quantile_90"}'
EOF
```

Register it in the placements:

```bash
oc patch cma multicluster-observability-addon --type=json -p '[
  {"op": "add", "path": "/spec/installStrategy/placements/0/configs/-", "value": {
    "group": "monitoring.rhobs",
    "resource": "scrapeconfigs",
    "name": "custom-recording-rule-metrics",
    "namespace": "open-cluster-management-observability"
  }}
]'
```

**Step 4 : Verify:**

After several minutes, query in Grafana Explore:

```promql
apiserver_request_duration_seconds:histogram_quantile_90
```

**Expected result: data from managed clusters.** The recording rule computes the 90th percentile locally on each managed cluster, and the result is federated to the hub.

## Summary

| Demo | Resource type | Where it runs | What it does | Difference from standard stack |
|---|---|---|---|---|
| 1 : Alert Rule | `thanos-ruler-custom-rules` ConfigMap | **Hub** (Thanos Ruler) | Evaluates alert expressions against centralized metrics | **None** : identical to standard stack |
| 2 : Custom metrics | `ScrapeConfig` CR (`monitoring.rhobs/v1alpha1`) | **Managed clusters** (PrometheusAgent) | Federates `kubelet_volume_stats_used_bytes` from managed clusters to hub | Must register in `ClusterManagementAddOn` placements. |
| 3 : Recording Rule | `thanos-ruler-custom-rules` ConfigMap | **Hub** (Thanos Ruler) | Computes `namespace:kubelet_volume_stats_used_bytes:sum` | **None** : identical to standard stack |
| 4 : Recording rule (managed cluster) | `PrometheusRule` CR (`monitoring.coreos.com/v1`) + `ScrapeConfig` CR | **Managed clusters** (Prometheus) + federation to hub | Computes `apiserver_request_duration_seconds:histogram_quantile_90` locally, federates result | Requires both a `PrometheusRule` (for the rule) and a `ScrapeConfig` (to federate the result). |

### MCOA key resources

| Resource type | Purpose |
|---|---|
| `ScrapeConfig` CR (`monitoring.rhobs/v1alpha1`) | Defines metrics to federate from managed clusters via `/federate` endpoint |
| `PrometheusRule` CR (`monitoring.coreos.com/v1`) | Defines managed-cluster-side recording rules |
| `PrometheusAgent` CR (`monitoring.rhobs/v1alpha1`) | Metrics collector (replaces legacy `metrics-collector-deployment` pod) |
| `thanos-ruler-custom-rules` ConfigMap | Hub-side alert/recording rules (unchanged from standard stack) |
| Auto-generated `ScrapeConfig` resources (`platform-metrics`, etc.) | Default platform metrics federation |
