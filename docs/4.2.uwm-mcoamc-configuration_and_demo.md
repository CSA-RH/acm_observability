
# User Workload Monitoring with MCOA

This document covers integrating OpenShift User Workload Monitoring (UWM) with ACM observability using the **Multicluster Observability Add-on (MCOA)**. It follows the same demo application as [4.1 UWM with standard metrics-collector](4.1.uwm-Standardmc-configuration_and_demo.md), but uses `ScrapeConfig` CRs instead of ConfigMap-based allowlists.

> **This document applies to the MCOA metrics collection stack** (PrometheusAgent + ScrapeConfig CRs). The standard and MCOA stacks are mutually exclusive — only one can be active at a time. If your environment uses the standard metrics-collector, see [4.1 UWM with standard metrics-collector](4.1.uwm-Standardmc-configuration_and_demo.md) instead.

---

## UWM on the managed cluster: two possible sources

Before forwarding UWM metrics to the hub, those metrics must first be collected on the managed cluster itself. There are two ways UWM can be configured locally, using the default UWM managed by the Cluster Monitoring Operator or using the ClusterObservability Operator.
This demo uses the **default OpenShift UWM** path. For the COO MonitoringStack variant, see the [COO MonitoringStack section](#note-coo-monitoringstack-as-uwm-source) at the bottom.

### Data flow: from application to hub

```
Managed Cluster                                Hub Cluster
┌──────────────────────────────────┐           ┌──────────────────────────────┐
│                                  │           │                              │
│  App (/metrics on port 8080)     │           │                              │
│        │                         │           │                              │
│        │ scraped by              │           │                              │
│        ▼                         │           │                              │
│  ServiceMonitor                  │           │                              │
│  (monitoring.coreos.com/v1)      │           │                              │
│        │                         │           │                              │
│        │ tells                   │           │                              │
│        ▼                         │           │                              │
│  prometheus-user-workload        │           │                              │
│  (local UWM Prometheus)          │           │                              │
│        │                         │           │                              │
│        │ /federate               │           │                              │
│        ▼                         │           │                              │
│  PrometheusAgent                 │           │  Observatorium API           │
│  (reads ScrapeConfig CRs         │           │  (gateway)                   │
│   monitoring.rhobs/v1alpha1)     │── push ──>│        │                     │
│                                  │ metrics   │        ▼                     │
│                                  │           │  Thanos Receive              │
│                                  │           │        │                     │
│                                  │           │        ▼                     │
│                                  │           │  Grafana / Thanos Ruler      │
│                                  │           │                              │
└──────────────────────────────────┘           └──────────────────────────────┘

Stage 1: Local scraping (same for standard and MCOA)
  App -> ServiceMonitor -> prometheus-user-workload

Stage 2: Forwarding to hub (MCOA-specific)
  prometheus-user-workload -> PrometheusAgent (via ScrapeConfig) -> hub
```

> **Note:** This diagram shows the default OpenShift UWM path (`prometheus-user-workload`). If the managed cluster uses a **COO MonitoringStack** instead, the local Prometheus is the MonitoringStack's own Prometheus instance, and the ScrapeConfig must manually specify its endpoint via `staticConfigs` (see the [COO MonitoringStack section](#note-coo-monitoringstack-as-uwm-source) at the bottom of this document).

## Prerequisites

- ACM 2.16 hub cluster with `MultiClusterObservability` CR created and observability enabled
- MCOA enabled (see [3.2 MCOA : section 2.1](3.2.mcoa-metricscollection_demo.md#21-mcoa-is-disabled-by-default))
- At least one managed cluster imported
- `oc` CLI access with cluster-admin

Enable MCOA if not already done.

---

## 1. Enable User Workload Monitoring on the managed cluster

On each managed cluster, enable UWM:

```bash
oc apply -f - <<'EOF'
apiVersion: v1
kind: ConfigMap
metadata:
  name: cluster-monitoring-config
  namespace: openshift-monitoring
data:
  config.yaml: |
    enableUserWorkload: true
EOF

# Verify UWM pods are running
oc -n openshift-user-workload-monitoring get pods
# Expected: prometheus-user-workload-0, prometheus-user-workload-1, thanos-ruler-user-workload-*
```

## 2. Deploy a sample user workload that exposes metrics

Deploy the same sample application used in the [standard stack demo](4.1.uwm-Standardmc-configuration_and_demo.md). The app (`prometheus-example-app`) serves an HTTP endpoint on port 8080 and exposes a `/metrics` path:

| Metric | Type | Description |
|---|---|---|
| `http_requests_total` | Counter | Total HTTP requests received, labeled by `method` and `code` |
| `http_request_duration_seconds` | Histogram | Duration of HTTP requests (with `_bucket`, `_count`, `_sum`) |
| `version` | Gauge | Application version info |

```bash
oc new-project demo-app

oc apply -f - <<'EOF'
apiVersion: apps/v1
kind: Deployment
metadata:
  name: demo-metrics-app
  namespace: demo-app
  labels:
    app: demo-metrics-app
spec:
  replicas: 1
  selector:
    matchLabels:
      app: demo-metrics-app
  template:
    metadata:
      labels:
        app: demo-metrics-app
    spec:
      containers:
        - name: metrics-exporter
          image: quay.io/brancz/prometheus-example-app:v0.5.0
          ports:
            - name: http-metrics
              containerPort: 8080
---
apiVersion: v1
kind: Service
metadata:
  name: demo-metrics-app
  namespace: demo-app
  labels:
    app: demo-metrics-app
spec:
  ports:
    - name: http-metrics
      port: 8080
      targetPort: 8080
  selector:
    app: demo-metrics-app
---
apiVersion: monitoring.coreos.com/v1
kind: ServiceMonitor
metadata:
  name: demo-metrics-app
  namespace: demo-app
spec:
  endpoints:
    - port: http-metrics
      interval: 30s
  selector:
    matchLabels:
      app: demo-metrics-app
EOF

oc expose svc demo-metrics-app
```

Verify the metrics are scraped by UWM:

```bash
oc -n openshift-user-workload-monitoring exec -it prometheus-user-workload-0 -- \
  promtool query instant http://localhost:9090 'version{namespace="demo-app"}'
```

## 3. Forward user workload metrics to the ACM hub

This is where the MCOA approach diverges from the standard stack. Instead of a ConfigMap on the managed cluster, create a `ScrapeConfig` CR **on the hub** with the `user-workload-metrics-collector` label.

**Step 1 : Create the ScrapeConfig on the hub:**

```bash
oc apply -f - <<'EOF'
apiVersion: monitoring.rhobs/v1alpha1
kind: ScrapeConfig
metadata:
  name: demo-app-uwm-metrics
  namespace: open-cluster-management-observability
  labels:
    app.kubernetes.io/component: user-workload-metrics-collector
spec:
  jobName: demo-app-uwm
  metricsPath: /federate
  params:
    match[]:
      - '{__name__="http_requests_total"}'
      - '{__name__="http_request_duration_seconds_bucket"}'
      - '{__name__="http_request_duration_seconds_count"}'
      - '{__name__="http_request_duration_seconds_sum"}'
      - '{__name__="version"}'
EOF
```

The `user-workload-metrics-collector` label tells the addon to scratch the metrics from the federated local Prometheus endpoing, on each managed cluster. 

**Step 2 : Register the ScrapeConfig in the ClusterManagementAddOn placements:**

Creating the ScrapeConfig alone is not enough, it must be registered in the `ClusterManagementAddOn` so it gets deployed to managed clusters:

```bash
# Inspect current placements:
oc get cma multicluster-observability-addon -o yaml | yq '.spec.installStrategy.placements'

# Register the ScrapeConfig:
oc patch cma multicluster-observability-addon --type=json -p '[
  {"op": "add", "path": "/spec/installStrategy/placements/0/configs/-", "value": {
    "group": "monitoring.rhobs",
    "resource": "scrapeconfigs",
    "name": "demo-app-uwm-metrics",
    "namespace": "open-cluster-management-observability"
  }}
]'
```

**Step 3 : Verify the ScrapeConfig was propagated to the managed cluster:**

```bash
# On the managed cluster, the deployment was done on the local-cluster:
oc get scrapeconfigs.monitoring.rhobs -n open-cluster-management-agent-addon | grep demo-app-uwm-metrics
```

## 4. Generate Traffic

```bash
ROUTE=$(oc -n demo-app get route demo-metrics-app -o jsonpath='{.spec.host}')
for i in $(seq 1 30); do curl -s "http://$ROUTE/" > /dev/null; done
echo "Sent 30 requests"
```

## 5. Verify user workload metrics on the hub

Generate some traffic and query the metrics in Grafana:

```bash
ROUTE=$(oc -n demo-app get route demo-metrics-app -o jsonpath='{.spec.host}')
curl -s "http://$ROUTE/"
```

After several minutes, query in Grafana Explore:

```promql
http_requests_total{namespace="demo-app"}
```

**Expected result: data appears.** on the Hub Grafana instance.

## Note: COO MonitoringStack as UWM source

If the managed cluster uses a Cluster Observability Operator (COO) `MonitoringStack` instead of the default OpenShift UWM, the ScrapeConfig must manually specify the target endpoint:

```yaml
apiVersion: monitoring.rhobs/v1alpha1
kind: ScrapeConfig
metadata:
  name: demo-app-uwm-metrics
  namespace: open-cluster-management-observability
  labels:
    app.kubernetes.io/component: user-workload-metrics-collector
spec:
  scrapeClass: ""
  scheme: HTTP
  staticConfigs:
    - targets:
        - my-monitoring-stack.my-monitoring-ns.svc:9090
  jobName: demo-app-uwm
  metricsPath: /federate
  params:
    match[]:
      - '{__name__="http_requests_total"}'
```

The `scrapeClass: ""` and `staticConfigs` override the default UWM Prometheus target with the COO MonitoringStack endpoint.

Source: [ACM 2.16 : Federating user workloads from COO](https://docs.redhat.com/en/documentation/red_hat_advanced_cluster_management_for_kubernetes/2.16/html-single/observability/index#federate-uw-coo)
