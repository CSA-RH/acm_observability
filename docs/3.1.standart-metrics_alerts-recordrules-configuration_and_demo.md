
# ACM Observability - Standart metrics collection: Custom Alerts, Recording Rules, and Metrics

## 1. Concepts

### 1.1 Data flow: how metrics move from managed clusters to the hub

```
Managed/Spoke Cluster                              Hub Cluster
┌──────────────────────────────┐             ┌────────────────────────────────────────┐
│  Prometheus                  │             │  Observatorium API                     │
│                              │             │(/api/metrics/v1/default/api/v1/receive)│
│  (platform + UWM)            │             │        │                               │
│        │                     │             │        ▼                               │
│        │ /federate           │             │  Thanos Receive                        │
│        │                     │             │                                        │
│        │                     │             │  Thanos Ruler                          │
│        ▼                     │             │  (evaluates hub-side alert &           │
│  metrics-collector           │── push ────>│   recording rules from                 │
│  (match flags = allowlist)   │             │   thanos-ruler-custom-rules)           │
│                              │             │                                        │
│  Alertmanager                │── alerts ──>│  Alertmanager                          │
│  (auto-configured)           │             │                                        │
└──────────────────────────────┘             └────────────────────────────────────────┘
```

Managed clusters alerts and metrics are sent to the hub:

1. **Metrics**: The `metrics-collector` on each managed cluster scrapes metrics from th spoke cluster local Prometheus and forwards them to the hub's Thanos. Only metrics in the **allowlist** are forwarded (default + custom).
2. **Alerts**: Fired alerts from each managed cluster's Alertmanager are forwarded to the hub's Alertmanager. This is automatic when observability is enabled, though if using the MCOA metrics collector, instead of the standant metrics collection, this has to be manually enabled.

### 1.2 Customization done using two ConfigMaps

All customization is done through two ConfigMaps in `open-cluster-management-observability` namespace:

| ConfigMap | Where rules execute | What you configure |
|---|---|---|
| `thanos-ruler-custom-rules` | **Hub** | Alert rules and recording rules that evaluate against the centralized metrics store. Has visibility across all clusters. |
| `observability-metrics-custom-allowlist` | **Managed clusters** | `names`: raw metrics to forward. `recording_rules`: PromQL expressions evaluated on each managed cluster's Prometheus, with results forwarded. |

## 2. Notes and Gotchas

### 2.1 Automatic alert forwarding from managed clusters

When observability is enabled, all platform alerts that fire on each managed cluster's Prometheus are forwarded to the hub's Alertmanager. If user workload monitoring is enabled, those alerts are forwarded too.

- `mco-disable-alerting: "true"` annotation on `MultiClusterObservability` CR : disables all alert forwarding
- `mco-disable-uwl-alerting: "true"` : disables only user workload alert forwarding

Source: [ACM 2.16 : Forwarding alerts](https://docs.redhat.com/en/documentation/red_hat_advanced_cluster_management_for_kubernetes/2.16/html-single/observability/index#forwarding-alerts). See also [The new Multi-Cluster Alerting UI: A developer preview](https://developers.redhat.com/articles/2025/03/27/new-multi-cluster-alerting-ui-developer-preview#).

## 3. Demos

### Demo 1: Custom Alert Rule on the Hub

Create a hub-side alert rule that fires when any cluster's CPU usage exceeds 5 cores. Uses `cluster:cpu_usage_cores:sum` which is in the default allowlist.

```bash
oc apply -f - <<'EOF'
kind: ConfigMap
apiVersion: v1
metadata:
  name: thanos-ruler-custom-rules
  namespace: open-cluster-management-observability
data:
  custom_rules.yaml: |
    groups:
      - name: cluster-health
        rules:
        - alert: ClusterCPUHealth-jb
          annotations:
            summary: Notify when CPU utilization on a cluster is greater than the defined utilization limit
            description: "The cluster has a high CPU usage: {{ $value }} core for {{ $labels.cluster }} {{ $labels.clusterID }}."
          expr: |
            max(cluster:cpu_usage_cores:sum) by (clusterID, cluster, prometheus) > 5
          for: 5s
          labels:
            cluster: "{{ $labels.cluster }}"
            prometheus: "{{ $labels.prometheus }}"
            severity: critical
EOF
```

Verify the rule is loaded : in Grafana Explore, query `ALERTS{alertname="ClusterCPUHealth-jb"}`.

```bash
oc get pods -n open-cluster-management-observability -w | grep "thanos-rule"
```

### Demo 2: Hub-side Recording Rule

Add a recording rule to the same ConfigMap as previous test `thanos-ruler-custom-rules`. This computes PV usage per namespace across clusters, using `kubelet_volume_stats_used_bytes`. Both alert and recording rules must coexist in a single `custom_rules.yaml`.

```bash
oc apply -f - <<'EOF'
kind: ConfigMap
apiVersion: v1
metadata:
  name: thanos-ruler-custom-rules
  namespace: open-cluster-management-observability
data:
  custom_rules.yaml: |
    groups:
      - name: cluster-health
        rules:
        - alert: ClusterCPUHealth-jb
          annotations:
            summary: Notify when CPU utilization on a cluster is greater than the defined utilization limit
            description: "The cluster has a high CPU usage: {{ $value }} core for {{ $labels.cluster }} {{ $labels.clusterID }}."
          expr: |
            max(cluster:cpu_usage_cores:sum) by (clusterID, cluster, prometheus) > 5
          for: 5s
          labels:
            cluster: "{{ $labels.cluster }}"
            prometheus: "{{ $labels.prometheus }}"
            severity: critical
      - name: namespace-volume-usage
        rules:
        - record: namespace:kubelet_volume_stats_used_bytes:sum
          expr: sum(kubelet_volume_stats_used_bytes) BY (namespace, clusterID, cluster)
EOF
```

Verify the rule was picked up:
```bash
oc exec -n open-cluster-management-observability observability-thanos-rule-0 -c thanos-rule -- curl -s http://localhost:10902/api/v1/rules | python3 -m json.tool | grep -A3 "namespace-volume-usage"
```

Query in Grafana Explore: `namespace:kubelet_volume_stats_used_bytes:sum`

**Expected result: no data.** The rule is healthy but `kubelet_volume_stats_used_bytes` is not in the default allowlist : it exists on each managed cluster's Prometheus but is not being forwarded from the managed clusters to the hub. Demo 3 fixes this.

### Demo 3: Custom Metrics Allowlist : fixing Demo 2

Add `kubelet_volume_stats_used_bytes` to the custom allowlist so it gets forwarded from managed clusters. Also define a managed-cluster-side recording rule that computes the API server request duration 90th percentile locally.

- `names` parameter: raw metrics to forward as-is. You list the metric name, and the metrics-collector adds it to its /federate match[] parameters, so the metric gets scraped from the local Prometheus and pushed to the hub.

- `recording_rules` parameter: PromQL expressions evaluated locally on each managed cluster's Prometheus, with the computed result forwarded to the hub. Useful for reducing cardinality (e.g., pre-aggregating a histogram into a percentile on the spoke, so only the single result is sent rather than all the raw buckets).

```bash
oc apply -f - <<'EOF'
kind: ConfigMap
apiVersion: v1
metadata:
  name: observability-metrics-custom-allowlist
  namespace: open-cluster-management-observability
data:
  metrics_list.yaml: |
    names:
      - kubelet_volume_stats_used_bytes
    recording_rules:
      - record: apiserver_request_duration_seconds:histogram_quantile_90
        expr: histogram_quantile(0.90, sum(rate(apiserver_request_duration_seconds_bucket{job='apiserver', verb!='WATCH'}[5m])) by (verb, le))
EOF
```

Verify the metrics collector restarts:
```bash
oc get pods -n open-cluster-management-observability | grep metrics-collector
```

After the restart (allow several minutes for ingestion), query Demo 2's metric again in Grafana Explore:

```promql
namespace:kubelet_volume_stats_used_bytes:sum
```

**Expected result: data appears.** The metric is now being forwarded, so the hub-side recording rule produces results. The managed-cluster-side recording rule result can also be queried:

```promql
apiserver_request_duration_seconds:histogram_quantile_90
```

## Summary

| Demo | ConfigMap | Where it runs | What it does |
|---|---|---|---|
| 1 : Alert Rule | `thanos-ruler-custom-rules` | **Hub** (Thanos Ruler) | Evaluates alert expressions against centralized metrics |
| 2 : Recording Rule | `thanos-ruler-custom-rules` | **Hub** (Thanos Ruler) | Computes `namespace:kubelet_volume_stats_used_bytes:sum` : shows no data until Demo 3 adds the input metric |
| 3 : `names` | `observability-metrics-custom-allowlist` | **Managed clusters** (metrics-collector) | Forwards `kubelet_volume_stats_used_bytes` from managed clusters to hub, enabling Demo 2 |
| 3 : `recording_rules` | `observability-metrics-custom-allowlist` | **Managed clusters** (Prometheus) | Computes `apiserver_request_duration_seconds:histogram_quantile_90` locally, forwards the result |
