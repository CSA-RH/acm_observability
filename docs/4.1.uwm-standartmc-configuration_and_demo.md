
# User Workload Monitoring with Standard Metrics Collection

This document covers integrating OpenShift User Workload Monitoring (UWM) with ACM observability **using the standard (legacy) metrics-collector**. It explains how to enable UWM on managed clusters, deploy a sample application with a ServiceMonitor, forward user workload metrics to the hub's Thanos, and create hub-side dashboards and alerts based on application-specific metrics.

> **This document applies to the standard metrics collection stack** (ConfigMap-based allowlists + `metrics-collector`). The standard and MCOA stacks are mutually exclusive — only one can be active at a time.

## User workload Monitoring (UWM) with ACM

UWM runs in each OpenShift cluster, in a separate Prometheus instance (`prometheus-user-workload`), that scrapes metrics from user namespaces via `ServiceMonitor` and `PodMonitor` CRs. 
- A `ServiceMonitor` selects Kubernetes Services by label and tells Prometheus which port and path to scrape: use it when your application exposes metrics through a Service. 
- A `PodMonitor` selects Pods directly by label, bypassing the need for a Service: use it for sidecar exporters, batch jobs, or pods that expose metrics but have no Service. 

ACM's observability collector can forward these user workload metrics to the hub's Thanos, making them available for centralized dashboarding and alerting.

Once on the hub, UWM metrics are handled identically to platform metrics : same Thanos store, same Grafana, same alert/recording rules:

| Aspect | Platform Metrics | User Workload Metrics |
|---|---|---|
| **Source Prometheus** | `prometheus-k8s` in `openshift-monitoring` | `prometheus-user-workload` in `openshift-user-workload-monitoring` |
| **Discovery** | Built-in OpenShift scrape targets | `ServiceMonitor` / `PodMonitor` CRs in user namespaces |
| **Allowlist ConfigMap key** | `metrics_list.yaml` | `uwl_metrics_list.yaml` |
| **ConfigMap location** | Hub (`open-cluster-management-observability`) : propagated to all clusters | Managed cluster (workload namespace or `open-cluster-management-addon-observability`) |


## 1. Enable User Workload Monitoring on the managed cluster

On each managed cluster (or the hub if self-managed), enable UWM:

```bash
oc apply -f - <<'EOF'
apiVersion: v1
kind: ConfigMap
metadata:
  name: cluster-monitoring-config
  namespace: openshift-monitoring
data:
  config.yaml: |
    enableUserWorkload: true
EOF

# Verify UWM pods are running
oc -n openshift-user-workload-monitoring get pods
# Expected: prometheus-user-workload-0, prometheus-user-workload-1, thanos-ruler-user-workload-*
```

## 2. Deploy a sample user workload that exposes metrics

Deploy a simple application in the Hub cluster that exposes Prometheus metrics. The app (`prometheus-example-app`) serves an HTTP endpoint on port 8080 and exposes a `/metrics` path with the following metrics:

| Metric | Type | Description |
|---|---|---|
| `http_requests_total` | Counter | Total number of HTTP requests received, labeled by `method` and `code` |
| `http_request_duration_seconds` | Histogram | Duration of HTTP requests in seconds (with `_bucket`, `_count`, `_sum` suffixes) |
| `version` | Gauge | Application version info |

The metrics are exposed via the standard Prometheus exposition format at `http://<pod-ip>:8080/metrics`. A `ServiceMonitor` CR is created alongside the app to tell `prometheus-user-workload` to scrape this endpoint every 30 seconds:

```bash
oc new-project demo-app 2>/dev/null || oc project demo-app

oc apply -f - <<'EOF'
apiVersion: apps/v1
kind: Deployment
metadata:
  name: demo-metrics-app
  namespace: demo-app
  labels:
    app: demo-metrics-app
spec:
  replicas: 1
  selector:
    matchLabels:
      app: demo-metrics-app
  template:
    metadata:
      labels:
        app: demo-metrics-app
    spec:
      containers:
        - name: metrics-exporter
          image: quay.io/brancz/prometheus-example-app:v0.5.0
          ports:
            - name: http-metrics
              containerPort: 8080
---
apiVersion: v1
kind: Service
metadata:
  name: demo-metrics-app
  namespace: demo-app
  labels:
    app: demo-metrics-app
spec:
  ports:
    - name: http-metrics
      port: 8080
      targetPort: 8080
  selector:
    app: demo-metrics-app
---
apiVersion: monitoring.coreos.com/v1
kind: ServiceMonitor
metadata:
  name: demo-metrics-app
  namespace: demo-app
spec:
  endpoints:
    - port: http-metrics
      interval: 30s
  selector:
    matchLabels:
      app: demo-metrics-app
EOF

oc expose svc demo-metrics-app
```

Verify the metrics are scraped by UWM:

```bash
# Check that the ServiceMonitor target is up:
# In the OpenShift console: Observe > Targets > Source: User

# Or query locally:
oc -n openshift-user-workload-monitoring exec -it prometheus-user-workload-0 -- \
  promtool query instant http://localhost:9090 'version{namespace="demo-app"}'
```

The sample app exposes metrics like `version`, `http_requests_total`, `http_request_duration_seconds_*`.

## 3. Forward user workload metrics to the ACM hub

Add the user workload metrics to the custom allowlist. For user workload metrics, create the `observability-metrics-custom-allowlist` ConfigMap **in the namespace of the workload** on the managed cluster (or in `open-cluster-management-addon-observability` for all namespaces):

```bash
# forward metrics from demo-app namespace:
oc apply -f - <<'EOF'
kind: ConfigMap
apiVersion: v1
metadata:
  name: observability-metrics-custom-allowlist
  namespace: demo-app
data:
  uwl_metrics_list.yaml: |
    names:
      - http_requests_total
      - http_request_duration_seconds_bucket
      - http_request_duration_seconds_count
      - http_request_duration_seconds_sum
      - version
EOF
```

> **Alternative:** To collect user workload metrics from **all namespaces** on the managed cluster, create the ConfigMap in `open-cluster-management-addon-observability` instead.

Wait for the metrics-collector to pick up the new allowlist and forward the metrics to the hub. The metrics-collector pod restarts when the ConfigMap changes, and then collects and forwards data on a ~5-minute interval (300s). Allow **up to 5 minutes** after the restart for the first data to appear in Grafana.

## 4. Generate Traffic

```bash
ROUTE=$(oc -n demo-app get route demo-metrics-app -o jsonpath='{.spec.host}')
for i in $(seq 1 50); do curl -s "http://$ROUTE/" > /dev/null; done
echo "Sent 10 requests"
```

On the hub cluster, query the forwarded metrics in Grafana:

```bash
#Generate one http request

```
